# Snapshot semantics for Pending Reviews

When a user starts a **Pending Review** against a mutating **Diff Source** (e.g. the working tree), the diff is captured as a **Diff Snapshot** and **Line Comments** anchor to that snapshot rather than to live file content. To re-examine after edits, the user explicitly "refreshes," which warns that anchored comments may not survive.

We chose this over GitHub-style content-anchoring (which re-locates comments across edits and marks them "Outdated") because robust anchoring is its own engineering project and the failure mode of bad anchoring is worse than the inconvenience of explicit refreshes. Snapshot semantics also match the agent-review workflow correctly: feedback is about the output the agent produced *at that moment*; if the agent iterates, the human starts a new review.

## Extension: Refresh on new PR commits (Guided PR Review)

For **Guided PR Review** the mutating **Diff Source** is the GitHub PR itself: when the author pushes new commits, the PR's head SHA moves. We detect this mid-session by polling the PR's current head SHA (on an interval and on window focus) against the SHA the session was built at, and surface a **banner** — never a silent change. Refresh stays explicit and, on top of re-snapshotting, also **regenerates the Guide** (a new head SHA misses the Guide cache and re-materializes the PR Worktree at the new SHA) and re-fetches the PR. The Guide is disposable/derived, so regenerating it is free of consequence; the **Pending Review** is the one durable, human-authored, non-regenerable state, so Refresh must preserve it.

Reconciliation rule (survivors vs. orphans): a **Line Comment** carries over iff every endpoint of its stored anchor (path + side + line number, plus the start endpoint for a multi-line range) resolves to **byte-identical content at the same position** in both the old and the new snapshot. Otherwise it is **orphaned** — kept and surfaced with a warning (the reviewer dismisses each once re-read/re-filed), never silently dropped or silently re-placed. Consistent with the no-re-location decision above, a comment whose content is unchanged but whose line number *shifted* is treated as an orphan: we never guess a new position. The reconciliation is a pure function (`src/shared/comment-reconciliation.ts`), unit-tested independently of the UI.
