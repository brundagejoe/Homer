# Installing Homer

Homer is a local, reviewer-side desktop tool for reviewing someone else's
GitHub PR. You launch it from your terminal, inside a repo, with a PR URL:

```sh
homer https://github.com/owner/repo/pull/123
```

This guide covers the **global install** — getting a `homer` command on your PATH
so you can run it from *any* repo, not just this cloned checkout. (For hacking
on Homer itself, use `bun run dev` / `bin/homer`; see the repo README.)

> Product name: **Homer**. Command: **`homer`** (ADR 0004).

## Prerequisites

- **macOS** (Apple Silicon). The packaged app targets arm64.
- **[`gh`](https://cli.github.com) authenticated** — Homer delegates all GitHub
  auth and API calls to the `gh` CLI. Run `gh auth login` once.
- **The `claude` CLI signed in on your subscription** — the review Guide is
  generated by your local `claude` CLI running on your existing Claude
  subscription. No API key is used or needed.
- **[`bun`](https://bun.sh)** — the installer builds Homer from source. It's the
  only build tool you install yourself: `scripts/install.sh` runs `bun install`
  (fetching `electron-vite`, `electron-builder`, etc.) and `bun run dist` for
  you. Needs network on the first build. (A prebuilt download is planned so this
  won't be required later.)

## One-command install

From a clone of this repo:

```sh
./scripts/install.sh
```

This:

1. builds the app (`bun run dist` → `dist/…/Homer.app`),
2. copies **`Homer.app` to `/Applications`**,
3. installs the **`homer`** shim to a PATH directory (`/usr/local/bin` if present,
   else `~/.local/bin` — it uses `sudo` only if the directory needs it), and
4. prints next steps (PATH + the Gatekeeper note below).

It's idempotent — re-run it any time to reinstall over a previous version.

## Using `homer` from any repo

```sh
cd ~/code/the-repo-the-pr-belongs-to
homer https://github.com/owner/repo/pull/123
```

Homer opens a single window with three tabs — **Activity · Guide · Diff** —
landing on Activity while the Guide generates in the background.

**How the repo is resolved.** Homer needs to know *which local repo* the PR
belongs to (that's where it fetches the head SHA, materializes the PR Worktree,
and reads `gh` auth). In the dev flow that repo is just the current directory.
But a globally-installed macOS `.app` launches with cwd `/`, **not** your repo —
so the `homer` shim captures your current directory and passes it to the app as a
`--repo=$PWD` launch flag next to the PR URL:

```sh
open -na "Homer" --args --repo="$PWD" "<pr-url>"
```

The app resolves its repo path as: `--repo=` flag → `DV_REPO` env var →
launch cwd (in that order — see `resolveRepoPath` in `src/main/launch.ts`).
So **run `homer` from inside the repo the PR belongs to.**

A second `homer <pr-url>` while Homer is already open focuses the window and points
it at the new PR. Known limitation: the *repo* of an already-open window is not
switched — quit Homer first if the new PR is in a different repo.

## Gatekeeper / unsigned app

The build is currently **unsigned and un-notarized** (ADR 0004). On first launch
macOS may warn that it "can't verify the developer." `install.sh` already clears
the quarantine flag on the copy it installs, but if you still get blocked:

- **right-click** `/Applications/Homer.app` → **Open** (once), or
- run `xattr -dr com.apple.quarantine /Applications/Homer.app`.

## Uninstall

```sh
./scripts/uninstall.sh
```

Removes `/Applications/Homer.app` and the `homer` shim (only one carrying Homer's
marker — it won't touch an unrelated `homer`). Cached PR Worktrees and Guides live
under `~/Library/Application Support/Homer` and can be deleted manually.

## Homebrew status

**Not published yet — the install script above is the recommended path today.**

A Homebrew **cask** would install the prebuilt `.app`:

```sh
brew tap brundagejoe/homer     # a separate homebrew-homer tap repo
brew install --cask homer
```

To make that real we'd need, **per release**:

- a **GitHub Release** hosting the `.dmg` at a stable URL,
- the dmg's **sha256** recorded in the cask, and
- a **tap** repository (`homebrew-homer` with `Casks/homer.rb`) to `brew tap`.

**Why not now:**

- **No releases exist.** A cask installs a *hosted, prebuilt* artifact; it does
  not build from source. Without a GitHub Release + versioned `.dmg` URL there's
  nothing for the cask to point at. Standing up releases (ideally CI-built,
  checksummed) is the actual prerequisite — the tap is the easy part.
- **The app is unsigned/un-notarized.** Homebrew doesn't remove the Gatekeeper
  prompt; a cask would still need a `--no-quarantine`-style workaround, i.e. no
  better than the install script here. Signing + notarizing (a separate
  follow-up) is what makes a cask genuinely nicer than the script.
- **Casks don't provide the `homer` command.** A cask only stages the `.app`. We'd
  still need a companion `binary` stanza or a separate formula to put `homer` on
  PATH — extra machinery the install script already handles in one place.
- **Maintenance cost.** Every release means bumping `version` + `sha256` in the
  tap (manually or via a bot). Not worth it for a pre-release, single-maintainer
  tool with no external users yet.

**Recommendation:** keep `scripts/install.sh` as the supported install. Revisit
Homebrew once (a) the app is code-signed + notarized and (b) there's a
CI-produced GitHub Release per version — at which point a cask becomes a real
upgrade. A ready-to-fill cask template lives at
[`packaging/homebrew/homer.rb`](../packaging/homebrew/homer.rb) (placeholders for
version / sha256 / URL) so adopting it later is a fill-in-the-blanks job.
